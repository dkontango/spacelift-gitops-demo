Two things that cause the most confusion

⚠️ One Spacelift session at a time. Spacelift allows only one active session per user (the login UI shows "Multi-session disabled"). Logging in via CLI or a second browser invalidates your web session. If an AI agent drives via its own browser, it will log you out — decide who holds the session before you start.

⚠️ Names are opaque and specific. Spacelift auto-generates stack/integration names like Prime Apollo 45 and Viking Terminal AWS — they tell you nothing about the repo or role. Always track the full tuple: (stack, repository, project root, IAM role ARN, integration name). The tutorial code uses exact literals: random_pet, aws_s3_bucket.orbit_storage, bucket_prefix = "orbit-storage-", role spacelift-orbit-labs-role.

The workflow, in one line

Push only to Forgejo → the push-mirror propagates to GitHub → Spacelift previews every PR → an OPA Plan policy blocks a public S3 bucket in the preview → merge deploys to AWS via keyless OIDC.

Why a vendor-agnostic tool for end-to-end CI/CD

The demo happens to use OpenTofu and AWS, but the reason to reach for Spacelift is that it doesn't lock you to either. Traditional pipelines were built for application code; infrastructure needs a system that understands state. As Spacelift puts it, "Traditional CI/CD doesn't work for infrastructure" — you want "workflows that understand state, not hacked-together jobs and scripts."

One orchestrator, every tool

Spacelift advertises "first-class support for Terraform, OpenTofu, CloudFormation, Pulumi, and Kubernetes" (plus Terragrunt and Ansible). The same PR-preview → policy → deploy loop shown here works whichever IaC you standardize on — no rewrite when the tool changes.

Understands your infrastructure

"Spacelift understands your stacks, dependencies, state, and resources, giving you one place to manage every part of your infrastructure pipeline." That's the difference between orchestration and a pile of CI jobs.

Policy as code, across the board

"Define OPA-based policies for plans, approvals, notifications, and security controls." The plan-block-public-s3 guardrail in this demo is exactly that — a Rego Plan policy that blocks a risky change in the PR preview.

Speed and control

Spacelift's own framing: "The speed developers demand. The control platform teams require." Self-service "Golden Paths via Blueprints, not Confluence pages", with "guardrails enforced by policy as code."

The headline, in Spacelift's words: "Ship infrastructure as fast as developers code" — "Spacelift fuses AI, IaC, and GitOps pipelines, so developers ship fast, and platform teams stay in control."

Quoted phrases above are Spacelift's own marketing language, from spacelift.io and its pricing page, cited here to keep this value proposition consistent with the vendor's. Drift detection and stack dependencies referenced in the demo are Starter+ features per that pricing page.
